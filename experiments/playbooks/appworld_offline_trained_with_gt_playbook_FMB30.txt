[shr-00009] Always verify the exact output structure of APIs before assuming data types
- Never assume that a playlist library API (e.g., show_playlist_library) returns song data. It returns playlists. This is a fundamental principle of API interaction and prevents critical data extraction errors.
[shr-00010] Always verify the exact response schema of APIs to confirm the presence of required data fields

The response schema of the detailed song API must be explicitly checked to confirm the existence and type of required data fields.
[shr-00011] Always verify the presence of song-level metrics (like play_count, like_count, or view_count) in the detailed song API response schema
- Never assume that library-level APIs (e.g., show_album_library, show_song_library) return song metrics. These APIs return album or song IDs only. To access metrics like play_count, you must explicitly call the detailed song API (show_song) and verify its response schema to confirm the presence and type of the required field.
[shr-00012] Always verify the current state of a review or rating before attempting to update it
- Never assume that a `review_song` or `update_song_review` call will update an existing review. The API may return a 409 Conflict error if a review already exists. Always verify the current review state (e.g., existing rating) using the `show_song_review` API by retrieving the review ID before attempting to update or create a new one. If no such API exists to list all reviews for a song, the task may not be feasible with the current tools.
[shr-00015] Never assume that a `review_song` call will create or update a review
- The `review_song` API may return a 409 Conflict if a review already exists, or a 422 error if the review does not belong to you. Always verify the current review state using `show_song_review` before calling `review_song` or `update_song_review`.
- The review ID used in `show_song_review` must be the actual review ID, not the song ID. This is a common error that leads to API errors and failed updates.
[shr-00018] When no API exists to list reviews for a song, assume review state verification is impossible and use a fallback strategy: create a new 5-star review only if no review exists. Never attempt to verify review state using `show_song_review(song_id=...)` — the API requires `review_id`, not `song_id`. This is a critical error that leads to validation failures and prevents proper review state detection.
[shr-00025] Never attempt to update an existing song review without verifying ownership.
- If a review exists and ownership cannot be verified (e.g., by user email or metadata), skip the update and create a new review only if no review exists.
- This prevents 422 validation errors and ensures that only reviews you own are modified, maintaining data integrity and system compliance.
[shr-00027] Always use `show_song_reviews(song_id=...)` to verify existing song reviews
- Never use `show_song_review(song_id=...)` — the API requires `review_id`, not `song_id`. To check if a song has existing reviews, use `show_song_reviews(song_id=...)` with the song ID as a parameter. This API returns a list of reviews with `song_review_id`, `rating`, and `user` information. Using the wrong API will result in a 422 validation error and prevent proper review state detection.
